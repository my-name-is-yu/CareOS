MemoryPath

Living Care Profile 設計 & CareOS v1 からの移行計画

Living Care Profile architecture & migration plan from CareOS v1

MemoryPath は Agents SDK で構築された GBrain を使い、断片化した認知症ケア記録から継続的に学習し、 すべてのケアギバーが数秒でパーソナライズされたケアを提供できる Living Care Profile を生成します。
MemoryPath uses GBrain, built with the Agents SDK, to continuously learn from fragmented dementia care records and generate a Living Care Profile that helps every caregiver provide personalized care in seconds.

1. 課題の再定義

1. Reframing the Problem

本質的な課題は「SOAPノートの要約」ではありません。何年分もの患者知識が数百のSOAPノートに埋もれていて、新しいケアギバーが毎回ゼロから始めなければならないことです。

MemoryPath は AI チャットボットではなく、看護師がどのみちやらなければならない仕事を AI が自動で準備し、看護師がレビューして承認するシステムです。次の看護師は数百のノートを読む代わりに、即座に以下を受け取ります。

The core problem is not "summarizing SOAP notes." It is that years of patient knowledge are buried across hundreds of SOAP notes, and every new caregiver has to start from scratch.

MemoryPath is not an AI chatbot. It is an AI system that automatically prepares the work nurses already have to do, while the nurse reviews and approves it. Instead of reading hundreds of notes, the next nurse instantly gets:

Who this resident is

その入居者がどんな人か — 経歴、性格、大切にしていること。

Background, personality, what matters to them.

最近の変化

What changed recently

Deviations from baseline, with citations to source records.

記録から学習した、有効な鎮静アプローチと既知のトリガー。

Calming approaches and known triggers learned from records.

Personalized care recommendations

「この入居者には何が効くか」の蓄積。医療判断ではなく運用上の支援。

Accumulated "what works for this resident." Operational support, not medical judgment.

シフト引き継ぎブリーフ

Auto-generated from the approved profile and recent records.

行動トレンドフラグ

Behavior trend flags

長期パターンの変化を検出し、逐語引用付きでフラグ。

Long-term pattern shifts, flagged with verbatim citations.

The central artifact shifts from a one-shot compile result to a continuously updated Living Care Profile. GBrain is promoted from an optional memory backend to the core reasoning layer of the product.

設計上の重要な決定

Key Design Decisions

  • 提案と承認の分離: 推論レイヤーはプロファイルを直接書き換えない。必ず「引用付きの更新提案」を出し、看護師の承認で初めて確定する。これが「チャットボットではない」ポジショニングの技術的な裏付けになる。
  • プロファイルは追記専用レコードの投影(projection): 元記録(CareRecord)は不変。プロファイルの全フィールドが元記録への引用を持ち、いつでも監査できる。
  • バージョン管理: プロファイルは承認のたびに新バージョン。「先週と今週で何が変わったか」自体がプロダクト価値(= 最近の変化)になる。
  • GBrain は知識ストア + 推論の両方: gbrain import でレコードを取り込み、search / think を Agents SDK エージェントのツールとして使う。既存の src/lib/gbrain.ts アダプタを拡張する。
  • フォールバック維持: GBrain が使えない環境でも JSON ストアだけでデモが動く、という v1 の設計原則を引き継ぐ。
  • Separation of proposal and approval: the reasoning layer never writes the profile directly. It always emits "proposed updates with citations," and only a nurse's approval commits them. This is the technical backbone of the "not a chatbot" positioning.
  • The profile is a projection of append-only records: source records (CareRecord) are immutable. Every profile field carries citations back to source records and is auditable at any time.
  • Versioning: each approval creates a new profile version. "What changed between last week and this week" is itself product value (= recent changes).
  • GBrain is both knowledge store and reasoning: ingest records via gbrain import, and use search / think as tools of Agents SDK agents. Extend the existing src/lib/gbrain.ts adapter.
  • Fallback preserved: keep the v1 principle that the demo works with the JSON store alone when GBrain is unavailable.

安全契約(v1 から継承し、継続学習向けに拡張)

Safety Contract (inherited from v1, extended for continuous learning)

  • 診断しない・処方しない・投薬量を提案しない・自律的なケア判断をしない。
  • プロファイル更新の全提案は元記録からの逐語引用を必須とし、検証で裏付けられない引用は削除する。
  • 看護師の承認なしにプロファイルは一切変わらない。
  • 危険な臨床表現には警告を出し、モデル出力を指示として扱わないよう促す。
  • No diagnosis, no prescriptions, no dosage suggestions, no autonomous care decisions.
  • Every proposed profile update requires verbatim citations from source records; unsupported citations are removed by verification.
  • The profile never changes without nurse approval.
  • Warn on unsafe clinical language and never treat model output as an order.

3. データモデル(コア)

3. Core Data Model

Zod スキーマとして src/lib/schema.ts に追加します(既存の CompileResult と並存)。

Added to src/lib/schema.ts as Zod schemas (alongside the existing CompileResult).

// All input sources normalize into one record shape
CareRecord {
  id: string
  residentId: string
  type: "soap_note" | "nurse_observation" | "family_memory"
      | "medication_record" | "incident_report"
  occurredAt: string          // ISO timestamp
  author: { role: string; name?: string }
  body: string                // original text — immutable, citation source
}

Citation {
  recordId: string
  quote: string               // verbatim — verified against record body
}

ProfileField<T> {
  value: T
  citations: Citation[]
  updatedInVersion: number
}

LivingCareProfile {
  residentId: string
  version: number
  approvedBy: string          // nurse identity
  approvedAt: string
  personSummary:        ProfileField<string>
  recentChanges:        ProfileField<Change[]>
  calmingApproaches:    ProfileField<string[]>
  knownTriggers:        ProfileField<string[]>
  careRecommendations:  ProfileField<Recommendation[]>
  handoffBrief:         ProfileField<string>
  trendFlags:           ProfileField<TrendFlag[]>
}

ProfileUpdateProposal {
  id: string
  residentId: string
  baseVersion: number         // version the diff applies to
  triggeredBy: string[]       // record IDs that triggered this proposal
  changes: FieldDiff[]        // per-field: before / after / citations / rationale
  status: "proposed" | "approved" | "rejected" | "edited_and_approved"
}

PHASE 1

データモデルとストア

Data Model & Stores

  • schema.ts に CareRecord / Citation / LivingCareProfile / ProfileUpdateProposal の Zod スキーマを追加。
  • data/records.json(追記専用)と data/profiles/(バージョン別 JSON)を新設し、ロード/保存関数を data.ts に実装。
  • 既存の history.json のノートを type: "soap_note" の CareRecord に、resident.json の memory を初期プロファイル(version 1、承認者 = seed)に移行するスクリプトを書く。
  • 合成データを拡充: 家族の記憶・服薬記録・インシデントレポートのサンプルレコードを追加(デモの説得力の核)。
  • Add Zod schemas for CareRecord / Citation / LivingCareProfile / ProfileUpdateProposal to schema.ts.
  • Create data/records.json (append-only) and data/profiles/ (versioned JSON) with load/save functions in data.ts.
  • Write a migration script: existing history.json notes become type: "soap_note" CareRecords; resident.json memory becomes the seed profile (version 1, approver = seed).
  • Expand synthetic data: sample family memories, medication records, and incident reports (the core of demo credibility).

完了条件: 移行スクリプト実行後、既存アプリが新ストア経由で従来どおり動く。

Done when: after running the migration script, the existing app works as before via the new stores.

PHASE 2

マルチソース取り込み

Multi-Source Ingestion

  • POST /api/records を新設: 5 種類のレコードタイプを受け付け、正規化してレコードストアに追記。
  • 追記時に GBrain へ自動インポートする同期処理を gbrain.ts に追加(GBrain 不在時はスキップしてフォールバック)。
  • UI にレコード入力フォームを追加(タイプ選択 + テキスト)。既存の NoteInput を汎用化。
  • Realtime 音声の書き起こしも nurse_observation レコードとして取り込めるようにする。
  • New POST /api/records: accepts the 5 record types, normalizes, and appends to the record store.
  • Add sync-to-GBrain on append in gbrain.ts (skip with fallback when GBrain is absent).
  • Add a record input form to the UI (type selector + text). Generalize the existing NoteInput.
  • Realtime voice transcripts can also be ingested as nurse_observation records.

完了条件: 5 タイプすべてのレコードを投稿でき、GBrain 検索でヒットする。

Done when: all 5 record types can be posted and are findable via GBrain search.

PHASE 3

推論レイヤー(GBrain × Agents SDK)

Reasoning Layer (GBrain × Agents SDK)

  • src/lib/agents/ に Agents SDK ベースのプロファイル更新パイプラインを実装。入力 = 新規レコード + 現行プロファイル + GBrain コンテキスト、出力 = ProfileUpdateProposal。
  • 役割別のエージェント/ステップ: 長期パターン抽出、ベースライン比較による変化検出(既存ドリフト検出を移植)、鎮静要因・トリガー学習、ケア推奨生成、トレンドフラグ生成。
  • GBrain の search / think をエージェントのツールとして公開。
  • 全提案を verify.ts(逐語引用検証)と lint.ts(危険表現)に通し、裏付けのない変更は提案から除外。
  • POST /api/proposals/generate(または records 追記時の自動トリガー)を実装。
  • Implement the Agents SDK profile-update pipeline in src/lib/agents/. Input = new records + current profile + GBrain context; output = ProfileUpdateProposal.
  • Role-specific agents/steps: long-term pattern extraction, change detection vs. baseline (port existing drift detection), calming/trigger learning, care recommendation generation, trend flag generation.
  • Expose GBrain search / think as agent tools.
  • Run every proposal through verify.ts (verbatim citations) and lint.ts (unsafe language); drop unsupported changes.
  • Implement POST /api/proposals/generate (or auto-trigger on record append).

完了条件: 新規レコード投稿 → 引用付きの更新提案が生成され、全引用が検証を通る。

Done when: posting a new record produces a cited update proposal whose citations all pass verification.

PHASE 4

レビュー & 承認 UI

Review & Approval UI

  • 提案一覧 + フィールドごとの before / after diff 表示。各変更に引用元レコードへのリンク。
  • 承認 / 却下 / 修正して承認 の 3 操作。承認で version + 1 の新プロファイルを保存。
  • POST /api/proposals/:id/approve / reject を実装。承認者と時刻を記録。
  • 「AI が準備し、看護師が承認する」という体験をこの画面で完成させる — ここがプロダクトの顔。
  • Proposal list + per-field before/after diff view. Each change links to its source records.
  • Three actions: approve / reject / edit-and-approve. Approval saves the profile as version + 1.
  • Implement POST /api/proposals/:id/approve / reject, recording approver and timestamp.
  • This screen completes the "AI prepares, nurse approves" experience — it is the face of the product.

完了条件: 提案を承認するとプロファイルの新バージョンができ、却下しても何も変わらない。

Done when: approving a proposal creates a new profile version; rejecting changes nothing.

PHASE 5

  • 引き継ぎブリーフ生成: 承認済みプロファイル + 直近レコードから生成(既存 compile の handoff brief を移植)。

  • デモシナリオ整備: 「新レコード投稿 → 提案生成 → 承認 → 次の看護師ビューが変わる」を 3 分で見せられる流れを作る。
  • "Next nurse" view: person summary, recent changes, calming/triggers, recommendations, and trend flags on one screen at a glance. Highlight changes between versions.

  • Switch the Realtime voice agent's instructions to the approved profile.
  • Build the demo scenario: "post record → proposal → approve → the next-nurse view changes," showable in 3 minutes.

完了条件: ワンラインピッチの体験(数秒でパーソナライズされたケアの把握)がデモで再現できる。

6. 依存関係と推奨順序

6. Dependencies & Recommended Order

Phase 1 → 2 → 3 → 4 → 5 は直列依存(それぞれ前段のデータ/API を使う)。Phase 0 はいつでも並行可能、Phase 6 は 5 の後。最短でデモ価値を出すなら:

  1. Phase 1 + 2 をまとめて(データ基盤。既存デモを壊さないことが唯一のリスク)
  2. Phase 3(ここが技術的な核。GBrain × Agents SDK の統合に最も時間を割く)
  3. Phase 4 + 5 をまとめて(体験の完成。デモの見せ場)
  4. Phase 0 と 6 は隙間時間で進める

Phases 1 → 2 → 3 → 4 → 5 are serially dependent (each consumes the previous phase's data/APIs). Phase 0 can run in parallel anytime; Phase 6 follows 5. For fastest demo value:

  1. Phase 1 + 2 together (data foundation; the only risk is not breaking the existing demo)
  2. Phase 3 (the technical core — spend the most time on the GBrain × Agents SDK integration)
  3. Phase 4 + 5 together (completing the experience — the demo highlight)
  4. Phases 0 and 6 fill the gaps
